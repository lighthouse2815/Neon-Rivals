EXECUTION RULES

Do not attempt to implement the entire project superficially in one pass.

Work through the phases below in strict order. A phase is complete only
after its acceptance criteria have been executed and verified.

Do not begin the next phase if the current phase has failing tests,
placeholder implementations, TODO comments, mocked core functionality,
or incomplete documentation.

At the end of every phase:

1. Run all relevant quality checks.
2. Start the application.
3. Test the implemented flow manually.
4. Update docs/05_PROGRESS.md with:
   - files created or changed
   - functionality completed
   - commands executed
   - test results
   - bugs found and fixed
   - remaining work
5. Continue automatically to the next phase when checks pass.

Do not stop to ask for confirmation unless blocked by credentials or an
external paid service.

Core game functionality must not contain:

- fake multiplayer
- simulated remote players presented as real players
- hardcoded successful responses
- placeholder buttons
- empty service methods
- TODO-only implementations
- skipped tests
- disabled TypeScript errors
- broad use of any
- duplicated Socket.IO contracts

Build a production-quality browser game named "Neon Duel" from the current

PRODUCT

Neon Duel is a real-time online 1v1 top-down arena shooter. Exactly two
players join a private room using a six-character room code or invitation
URL. Each player can move, aim, shoot, dash, take damage, and compete in a
best-of-five match. The first player to win three rounds wins the match.

Use original visual assets only. Do not use copyrighted game characters,
logos, music, or copied assets.

TECH STACK

Create a pnpm and Turborepo monorepo:

- apps/client: Phaser 3, TypeScript, Vite, Socket.IO Client, Howler.js
- apps/server: Node.js, TypeScript, Fastify, Socket.IO, Zod
- packages/shared: shared event contracts, schemas, constants, and types
- Vitest for unit and integration tests
- Playwright for two-browser multiplayer E2E tests
- ESLint and Prettier

Do not add a database to the MVP. Store rooms and matches in server memory.
Design the code so PostgreSQL and Redis can be introduced later without
rewriting the game domain.

MULTIPLAYER ARCHITECTURE

Use a server-authoritative architecture.

The client must send input commands only. The server owns and validates:

- player positions
- movement speed
- projectiles
- hit detection
- health and damage
- shooting cooldown
- dash cooldown
- round state
- match score
- spawn points
- match winner

Run the server simulation at 30 ticks per second and send snapshots at
approximately 20 snapshots per second.

Implement:

- normalized player input
- input sequence numbers
- snapshot interpolation for remote players
- basic local-player prediction
- server reconciliation
- ping measurement
- rate limiting
- payload validation
- maximum message size
- stale-input rejection

ROOM FLOW

Implement:

- create private room

- join room
- maximum two players
- ready state

- leave room

- heartbeat

- reconnect token
- 15-second reconnect grace period
- prevent a third player from taking a disconnected player's slot
- pause the round temporarily while waiting for reconnection
- award the round to the connected opponent after reconnect timeout

GAMEPLAY

Controls:

- WASD or arrow keys: move
- mouse: aim
- left click: shoot
- Space: dash
- Escape: pause/settings overlay

Rules:

- 100 health per player
- one projectile weapon
- configurable damage and fire cooldown
- configurable dash cooldown
- one symmetrical arena
- best of five rounds
- first to three round wins
- short countdown before each round
- spawn protection for a brief configurable duration
